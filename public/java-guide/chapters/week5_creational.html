1. Understanding Design Patterns

Design patterns are typical solutions to common problems in software design. They are like blueprints that you can customize to solve a particular design problem in your code.

Creational Patterns provide object creation mechanisms that increase flexibility and reuse of existing code.

2a. Singleton Pattern

Goal: Ensure a class has only one instance and provide a global point of access to it.

Use Case: Database connections, Logging, Configuration settings.

public class Database { // 1. Static variable to hold the single instance private static Database instance; // 2. Private constructor to prevent instantiation private Database() { System.out.println("Connecting to SQL..."); } // 3. Public static method to get the instance public static Database getInstance() { if (instance == null) { instance = new Database(); } return instance; } }
Private Constructor: Crucial! No one outside can say 'new Database()'.
Lazy Initialization: The object is created only when 'getInstance()' is called for the first time.

2b. Factory Pattern

Goal: Define an interface for creating an object, but let subclasses decide which class to instantiate.

Use Case: A logistics app that handles different types of transport (Truck, Ship, Plane).

interface Transport { void deliver(); } class Truck implements Transport { public void deliver() { System.out.println("By road"); } } class Ship implements Transport { public void deliver() { System.out.println("By sea"); } } class LogisticsFactory { public Transport createTransport(String type) { if (type.equals("ROAD")) return new Truck(); if (type.equals("SEA")) return new Ship(); return null; } }
Factory Logic: The client doesn't need to know the complex logic of creating objects. It just asks for "ROAD" or "SEA".

2c. Builder Pattern

Goal: Construct a complex object step by step.

Use Case: Building a custom Pizza or a Computer with many optional configurations.

class Pizza { String dough; String sauce; String topping; } class PizzaBuilder { private Pizza pizza = new Pizza(); public PizzaBuilder setDough(String d) { pizza.dough = d; return this; // Chaining } public PizzaBuilder setSauce(String s) { pizza.sauce = s; return this; } public Pizza build() { return pizza; } } // Usage Pizza p = new PizzaBuilder().setDough("Thin").setSauce("Tomato").build();
Method Chaining: Returning 'this' allows us to chain calls like .set().set().build().

Seminar: Identifying Patterns

a. From User Stories to Patterns

Story: "As a system admin, I want to ensure only one connection to the printer exists so that jobs don't get mixed up."

Pattern: Singleton (Only one instance).

Story: "As a user, I want to create different types of documents (PDF, Word, HTML) without knowing the internal generation logic."

Pattern: Factory (Hides creation logic).

b. Singleton Model (2D/3D Analogy)

Imagine a Sun in a solar system model.

  • 2D Model: A drawing on paper. You can draw multiple suns, but that's physically wrong for our solar system.
  • 3D Singleton Model: A physical model where you have ONE plastic ball representing the Sun. All planets revolve around this ONE ball. You cannot just "add another sun" to the center. The structure enforces singularity.

Lab 5: Implementing Creational Patterns

Task 1: Singleton

Create a `ConfigurationManager` class. It should store settings (key-value pairs). Ensure only one manager exists in the app.

Task 2: Factory

Create a `NotificationFactory`. It should create `EmailNotification`, `SMSNotification`, or `PushNotification` objects based on user input.

Task 3: Builder

Create a `ComputerBuilder`. Allow users to build a PC with optional components: `setRAM()`, `setGPU()`, `setStorage()`. Finally call `build()`.
